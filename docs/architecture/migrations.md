# Database migration strategy

> How Prumo manages schema changes â€” calibrated for AI-assisted
> development on top of Postgres + SQLAlchemy 2.0 async + Alembic +
> Supabase. Read this before writing your first migration on the project.

## TL;DR

- **One migration system per concern.** Alembic owns `public.*` tables.
  Supabase migrations own `auth.*` and `storage.*`. Never mix.
- **Hand-write migrations**, don't trust `--autogenerate` blindly. The
  docstring carries *why*; the SQL carries *what*. Both matter for AI
  agents reading future regressions.
- **One logical change per migration.** Coupled changes get split.
- **Squash on stable releases**, not before. The 18-migration trail from
  the 2026-04 HITL refactor stays as-is until v1.0 ships; the next major
  cycle gets a fresh baseline.
- **Tests use real Postgres + real `alembic upgrade head`**. No
  in-memory shortcuts â€” schema bugs need to surface in CI.

## Two migration systems, sharply scoped

| System | Scope | Where | Tool |
|---|---|---|---|
| **Alembic** | Application schema (`public.*` tables, indexes, triggers, RLS, enums) | `backend/alembic/versions/` | `uv run alembic` |
| **Supabase migrations** | Auth + Storage built-ins (`auth.*`, `storage.buckets`, bucket policies) | `supabase/migrations/` | `supabase migration new` |

If you're touching a SQLAlchemy model in `backend/app/models/`, you must
use **Alembic**. If you're creating a storage bucket or an auth policy,
you must use **Supabase**. The rule is binary â€” there is no overlap zone
to negotiate.

## Migration anatomy

Every migration file follows this shape:

```python
"""<one-line imperative summary>

Revision ID: YYYYMMDD_NNNN
Revises: YYYYMMDD_NNNN-1
Create Date: YYYY-MM-DD
"""

from alembic import op

revision = "YYYYMMDD_NNNN"
down_revision = "YYYYMMDD_NNNN-1"


def upgrade() -> None:
    """1-3 paragraph rationale: WHY this change exists, what invariant
    it establishes, and what it interacts with. Future you (or a future
    AI agent) reads this docstring before reading the SQL â€” so the
    *why* must be self-contained."""
    op.execute("...")


def downgrade() -> None:
    op.execute("...")
```

**Why hand-written, not autogenerated:**

- `--autogenerate` drops index renames, comment changes, and constraint
  reordering as deletions+creations. Output is noisy; signal is low.
- It can't capture *why*, only *what*. Reviewing AI-generated code is
  much faster when the docstring tells you the intent in advance.
- Some changes (RLS policies, function bodies, complex CHECKs) aren't
  autogenerated at all.

When `--autogenerate` is genuinely useful (large, mostly-mechanical
schema diff after refactoring many models), use it as a *draft* â€” review
every line, rewrite the docstring, split into one-change-per-file.

## Naming + ordering

- File name: `YYYYMMDD_NNNN_<short_kebab_summary>.py`. Date is the day
  the migration was *written*; NNNN is the next sequential number across
  the whole repo (no per-day reset).
- `revision` matches the date+number prefix.
- `down_revision` points at the previous head â€” never branch.

## When to squash

> **2026-04-28 squash already happened.** The original 18 migrations
> (now in `backend/alembic/versions/archive/`) were squashed into a
> single baseline `0001_baseline_v1.py` + `baseline_v1.sql`. New work
> stacks on top.

The trigger then was: 18 migrations totalling ~5,800 LOC, including 612
LOC of `0008_unified_evaluation_model_skeleton` that `0016_drop_008_stack`
fully reverted. A fresh dev DB had to execute ~5,800 LOC to land at a
schema that ~2,000 LOC of squash baseline produces in one shot.

**Squash rule of thumb:** when the project ships a stable release (v1.0,
v2.0, â€¦) and the in-flight refactors have settled, replace the
predecessor migrations with a single baseline that captures the current
schema. Document the squash in the new baseline's docstring + a row in
the architecture doc.

**Don't squash:**
- During an active refactor cycle (you'll just have to re-squash).
- If production is reading from the affected tables and you're not
  prepared to `alembic stamp` carefully.
- For "cleanup" alone â€” every squash invalidates someone's local dev DB.

**Squash recipe** (the 2026-04-28 squash followed this exactly â€” keep
this as the template for the next squash):

1. Pick a baseline id. Convention: `0001_baseline_v<N>` where N is the
   schema generation (we used `0001_baseline_v1`).
2. Capture current schema:
   ```bash
   supabase db dump --local --schema=public --file=/tmp/dump.sql
   ```
3. Strip the noise:
   ```bash
   awk '
   /^SET / { next }
   /^SELECT pg_catalog\.set_config/ { next }
   /^ALTER (TABLE|TYPE|FUNCTION|SCHEMA) .* OWNER TO/ { next }
   /^CREATE TABLE IF NOT EXISTS "public"\."alembic_version"/,/^);/ { next }
   /^ALTER TABLE ONLY "public"\."alembic_version"/,/PRIMARY KEY \("version_num"\);/ { next }
   /^GRANT ALL ON TABLE "public"\."alembic_version"/ { next }
   { print }
   ' /tmp/dump.sql > backend/alembic/versions/baseline_v<N>.sql
   ```
4. Create the Python wrapper (`0001_baseline_v<N>.py`) that reads + executes
   the `.sql` file. Pattern:
   ```python
   from pathlib import Path
   from alembic import op
   revision = "0001_baseline_v<N>"
   down_revision = None

   _SQL = Path(__file__).parent / "baseline_v<N>.sql"

   def upgrade() -> None:
       op.execute(_SQL.read_text(encoding="utf-8"))

   def downgrade() -> None:
       op.execute("DROP SCHEMA IF EXISTS public CASCADE;")
       op.execute("CREATE SCHEMA public;")
   ```
5. Move all old migrations: `git mv backend/alembic/versions/*.py
   backend/alembic/versions/archive/` (Alembic ignores subdirectories
   under `versions/`).
6. Stamp existing DBs at the new baseline:
   ```bash
   uv run alembic stamp --purge 0001_baseline_v<N>
   ```
   The `--purge` flag clears any stale revision row that points at an
   archived id. Verify with `uv run alembic current`.
7. Run the full test suite. If tests pass, the schema is functionally
   identical to what the migration trail produced.

**Note on `alembic check`:** after a squash, `alembic check` may report
`modify_type` diffs for `String()` vs `TEXT()` and similar SQLAlchemy
naming-convention noise. Those are pre-existing model/DB quirks
unmasked by the squash, not real differences. Trust the test suite,
not `alembic check`, for "is the schema right?".

## Tests

Integration tests run against a real local Postgres (Supabase via
`supabase start`). They expect the schema produced by `alembic upgrade
head` â€” every migration is implicitly exercised on every CI run.

Coverage tiers we care about:

- **Migration-specific tests**: assert the migration produced the
  expected schema (column nullability, constraint names, index
  presence). Examples: `tests/integration/test_evidence_evolution.py`,
  `tests/integration/test_kind_discriminator.py`,
  `tests/integration/test_synthetic_runs_migration.py`.
- **Lifecycle tests** that exercise the migration's *intent*. E.g.
  `test_run_lifecycle_lazy_creates_v1_for_template_without_version`
  validates the lazy-create path the migration relied on.

When you add a migration that introduces a non-trivial invariant, add a
test for the invariant. It makes both production safer and future AI
edits easier â€” they have something concrete to break.

## Make targets

```bash
make db-fresh      # reset + migrate + seed â€” the AI-friendly dev cycle
make db-setup      # reset + migrate (no seed)
make db-migrate    # alembic upgrade head only
make db-rollback   # alembic downgrade -1
make db-history    # alembic history --verbose
make db-current    # alembic current
make db-generate MSG="..."  # autogenerate (use as draft, then rewrite)
make db-seed       # python -m app.seed (idempotent)
```

Use `make db-fresh` after pulling main / after a structural refactor /
before running an integration test suite that touches schema. It's
idempotent w.r.t. seed and gives you a clean reproducible baseline in
one command.

## RLS in migrations

When a migration creates a table that contains user-scoped data, it
must:

1. `ALTER TABLE ... ENABLE ROW LEVEL SECURITY;`
2. Create explicit `SELECT` / `INSERT` / `UPDATE` / `DELETE` policies in
   the same migration â€” never a follow-up.
3. Reference `is_project_member` / `is_project_reviewer` /
   `is_project_manager` (defined in earlier migrations as `SECURITY
   DEFINER` functions). Mirror the existing policy shape.

See `0012_workflow_tables.py` and `0018_is_project_reviewer_rls.py` for
the canonical patterns.

## When AI assistants edit migrations

- **Never** edit a committed-and-deployed migration in place. Add a new
  migration that fixes the issue.
- **Never** rename a `revision` id once the migration is on someone
  else's machine â€” `down_revision` chains break silently.
- When in doubt, write the migration's downgrade first. If you can't
  write a downgrade, you don't fully understand the upgrade.

## Recommended next steps (not yet adopted)

These would further cut friction for AI-assisted dev. Each is a small,
isolated improvement:

- **Squawk in CI** â€” Postgres migration linter; flags
  `ADD COLUMN NOT NULL` without default, drops on populated tables, FKs
  without indexes, and other dangerous patterns. Two-line GitHub
  Actions step on top of `make db-fresh`. https://github.com/sbdchd/squawk
- **Atlas (evaluation only)** â€” `atlasgo.io` does declarative
  schema-as-code with Postgres + SQLAlchemy integration. Agent edits a
  single source-of-truth file; Atlas computes the migration. Worth
  evaluating in a sandbox before v2.0 â€” adoption cost ~1 sprint, payoff
  is highest in refactor-heavy AI dev cycles.
- **Supabase Branching** â€” ephemeral DB per PR for migration safety.
  Solo dev makes this premature; consider when team grows.
