# Accessibility patterns

Radix gives us most of the keyboard + focus + ARIA primitives. This file
covers the gaps Radix cannot fill on its own, plus prumo-specific patterns
(extraction streaming, HITL decision changes).

## What Radix already handles

If you reach for Radix-backed primitives (`Dialog`, `Popover`, `Select`,
`DropdownMenu`, `Tabs`, `Accordion`, `Tooltip`, `Toggle`, `Slider`, `Switch`,
`AlertDialog`, `NavigationMenu`), you get for free:

- Focus trap inside overlays.
- Focus return to trigger on close.
- Esc to close.
- Arrow-key navigation in lists.
- `aria-expanded`, `aria-controls`, `aria-haspopup`, `role` attrs.
- `inert` on background content when an overlay is open.
- Correct ID associations for label/description.

Do not re-implement these. If you build a custom select with `<div>`s and
`onKeyDown`, you will get it wrong.

## What you still have to do

### Icon-only buttons

`<Button variant="ghost" size="icon"><X /></Button>` is unlabelled. Add one:

```tsx
<Button variant="ghost" size="icon" aria-label="Close panel">
  <X aria-hidden="true" />
</Button>
```

`aria-hidden` on the icon stops the screen reader from announcing "x mark" on
top of the label. Lucide icons are decorative by default but it never hurts.

### Form inputs

The shadcn `Form*` family in `ui/form.tsx` wires `aria-invalid`,
`aria-describedby`, label associations, and message IDs automatically — *if*
you use `FormField` / `FormItem` / `FormLabel` / `FormControl` / `FormMessage`.
Plain `<input>` skips all of it.

```tsx
<FormField
  control={form.control}
  name="dose"
  render={({ field }) => (
    <FormItem>
      <FormLabel>Dose (mg)</FormLabel>
      <FormControl>
        <Input type="number" {...field} />
      </FormControl>
      <FormDescription>Per administration, not daily total.</FormDescription>
      <FormMessage />  {/* announces zod errors when present */}
    </FormItem>
  )}
/>
```

For inputs outside react-hook-form, do the wiring by hand:

```tsx
<Label htmlFor="ext-pmid">PMID</Label>
<Input
  id="ext-pmid"
  aria-describedby={hasError ? "ext-pmid-error" : undefined}
  aria-invalid={hasError || undefined}
/>
{hasError && <p id="ext-pmid-error" className="text-xs text-destructive">{error}</p>}
```

### Live regions for async state

Extraction streaming updates fields without user interaction. Screen readers
will not announce them unless the parent is a live region.

```tsx
<section aria-live="polite" aria-atomic="false">
  {fields.map((f) => <FieldRow key={f.id} field={f} />)}
</section>
```

- `aria-live="polite"` for routine updates ("Field 'dose' updated").
- `aria-live="assertive"` only for errors or blocking conditions; it
  interrupts whatever the user is reading.
- `aria-atomic="false"` means "only re-announce the changed subtree", not the
  whole region.

For toasts: trust `ui/toaster.tsx`. It already wraps a live region. Do not
nest another one.

### Status badges and pills

A colored pill is not enough — color cannot be the only signal.

```tsx
<StatusPill tone="approved">
  <Check aria-hidden="true" />
  <span>Approved</span>
</StatusPill>
```

A screen reader user gets "Approved" regardless of theme; a dichromatic user
gets the check icon. Both signal the same state.

### Required fields

```tsx
<Label htmlFor="title">
  Title
  <span aria-hidden="true" className="text-destructive">*</span>
</Label>
<Input id="title" required aria-required="true" />
```

The asterisk is visual; the `required` + `aria-required` attributes are
programmatic.

### Skip links

The `LayoutShell` (or whichever component wraps the page) should expose a
skip-to-content link. If you find a page without one, add:

```tsx
<a
  href="#main"
  className="
    sr-only focus:not-sr-only focus:absolute focus:top-4 focus:left-4
    focus:z-50 focus:px-4 focus:py-2 focus:bg-background focus:border
    focus:rounded-md focus:shadow-md
  "
>
  Skip to main content
</a>
<main id="main">...</main>
```

`sr-only` hides visually; `focus:not-sr-only` brings it back when the user
tabs into it. The class is defined by Tailwind out of the box.

### Reduced motion

Any keyframe animation must guard against `prefers-reduced-motion: reduce`.
The pattern lives at the top of `frontend/index.css` (`field-just-updated`):

```css
@keyframes field-just-updated-fade { /* ... */ }
.field-just-updated { animation: field-just-updated-fade 1.5s ease-out forwards; }
@media (prefers-reduced-motion: reduce) {
  .field-just-updated { animation: none; }
}
```

For Tailwind utilities, the equivalent is `motion-reduce:transition-none
motion-reduce:animate-none`. Apply on any element with a non-trivial
transition or animation.

## Color contrast

The semantic tokens **on their own backgrounds** are WCAG AA:

- `text-foreground` on `bg-background` — AAA.
- `text-muted-foreground` on `bg-background` — AA.
- `text-muted-foreground` on `bg-muted` — **fails AA**. Use `text-foreground`
  for body copy that lands on muted surfaces; reserve `text-muted-foreground`
  for non-essential text on `bg-background` / `bg-card`.
- `text-primary-foreground` on `bg-primary` — AAA.
- `text-destructive-foreground` on `bg-destructive` — AAA.

When in doubt, test in DevTools (contrast inspector on the Accessibility tab)
or via the Lighthouse audit.

## Focus indicators

Every interactive element must show focus. The standard ring is:

```
focus-visible:outline-none
focus-visible:ring-2
focus-visible:ring-ring
focus-visible:ring-offset-2
```

The base `ui/button.tsx` already includes this. If you add a new clickable
that is **not** a Button (e.g. a row, a card, a tab in a custom tablist), copy
the same four utilities.

Never disable focus with `focus:outline-none` alone — that strips the system
default without replacement. Use `focus-visible:outline-none` plus a custom
ring.

## Keyboard reachability

- All interactive elements must reach by Tab.
- The tab order must follow visual reading order. If the DOM is wrong, fix
  the DOM, not `tabIndex`.
- Modal-like overlays must trap focus (Radix Dialog does, custom popovers may
  not — verify).
- `tabIndex={-1}` is fine for elements that should not be in the tab order
  but might be focused programmatically (e.g. a heading inside a freshly
  opened panel to anchor the screen reader).

## Testing flow

Before merging UI changes:

1. **Tab through the new flow.** Every interactive element receives focus and
   shows a ring.
2. **Open it with a screen reader on** (VoiceOver: `Cmd+F5`). Form labels
   should announce; toasts should pop; live regions should fire.
3. **Toggle to dark mode.** Re-check contrast.
4. **Browser zoom to 200%.** Nothing should clip; horizontal scroll only on
   intentional overflow areas (PDFs, wide tables).
5. **DevTools → Rendering → Emulate `prefers-reduced-motion: reduce`.**
   Animations should soften or disappear.

## Specific prumo flows worth a second look

- **Extraction row hover actions.** Visible-on-hover is fine for sighted
  users; keyboard users get them via Tab anyway because we set `opacity` not
  `display: none`. Verify the row itself is focusable (button-like role) if
  the row is clickable.
- **HITL decision change.** When the reviewer publishes a decision, the
  resulting status pill change must announce via a `polite` live region.
- **PDF viewer.** The viewer chrome is mostly buttons; the PDF itself is
  decorative for AT users — provide a "View as text" or "Download PDF" fallback
  near the chrome.
- **Comparison view.** The two columns must read in order: "AI proposal,
  field X, value Y" → "Reviewer decision, field X, value Z". Aria-label the
  panels.
